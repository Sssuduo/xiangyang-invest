## 1. 背景与动机

### 1.1 问题

录音识别/总结/校正/查看功能最初与「活动台账」模块紧耦合（`admin_activity_ledger_audio.py` 870 行）。V15.4 迁移到「招商动态」时，采用了**重写**而非复用，导致：

- 两套独立代码维护成本高
- 招商动态版缺失大量能力（术语校正、三版本 Tabs、Word 下载、LLM 选择）
- 重写引入多个 bug（async 健康检查、progress 文案错误、错误处理缺失）

### 1.2 设计目标

- **组件化**：录音是跨模块基础能力，所有模块通过统一 service 层调用
- **零重写**：成熟代码原样复用，不改变经过验证的逻辑
- **解耦**：路由层（API 前缀）与 service 层（业务逻辑）分离
- **可扩展**：未来模块（招商动态、 future modules）仅需注册字段映射即可接入

## 4. `services/audio_service.py` 详细设计

### 4.1 字段注册表

```python
_FIELD_REGISTRY: dict[type, dict[str, str]] = {}

def register_audio_model(model_class: type, fields: dict[str, str]):
    """注册模型字段名映射"""
```

每个模型在模块加载时调用一次 `register_audio_model()`，声明其音频字段名：

```python
_LEDGER_AUDIO_FIELDS = {
    "files": "audio_files",
    "status": "audio_status",
    "message": "progress_message",
    "pct": "progress_pct",
    "duration": "audio_duration",
    "transcript": "audio_transcript",
    "summary": "audio_summary",
    "segmented": "audio_transcript_segmented",
    "clean": "audio_transcript_clean",
    "structured": "audio_summary_structured",
    "docx_path": "audio_docx_path",
    "docx_size": "audio_docx_size",
    "archive": "audio_archive",
    "archive_size": "audio_archive_size",
    "model_id": "summary_model_id",
}
register_audio_model(ActivityLedger, _LEDGER_AUDIO_FIELDS)
```

**优势**：未来新增模型（如 `ConstructionProject`）只需注册自己的字段映射即可复用全部能力。

### 4.2 取消控制

```python
_CANCEL_EVENTS: dict[tuple[type, int], threading.Event] = {}

def is_task_cancelled(model_class: type, item_id: int) -> bool:
    """内存注册表 + DB 双源检查"""

    """注册取消事件"""
```

- 键控 `(model_class, item_id)`，不同模型的取消事件互不干扰
- 双源检查：优先内存注册表，回退 DB `audio_status == 'cancelled'`

### 4.3 audio_files 读写

```python

def set_audio_files(item, files_list: list)
def resolve_audio_path_by_url(url: str) -> str
```

通过 `_fields(item)` 自动获取字段名，无需硬编码。

### 4.4 核心 Worker

#### `run_async_processing(app, model_class, item_id)`

后台线程：ASR 转写 → 实时进度 → 自动总结。

关键行为：
- 检查 ASR 服务健康（`check_asr_health()`），不可用立即失败
- 已识别段（`status == 'ok'`）保留不覆盖（`_transcript_cache`）
- 每 30 秒切片为一个进度单位，实时计算预估剩余时间
- 取消检查（段间隙退出）
- ASR 完成后自动触发结构化总结

#### `run_summary_only(app, model_class, item_id, model_id=None)`

仅重新总结（基于现有转写内容），不跑 ASR。

#### `apply_summary_to_item(item, full_text, model_id=None)`

生成三版本结构化总结（segmented + clean + structured）+ docx 文件。

### 4.5 内部辅助

## 11. 风险与限制

1. ~~**`admin_activity_audio.py` 尚未改为复用**~~ ✅ 已改造为薄包装复用 `audio_service`
2. **文本校正**：ID 兼容性问题未解决（`TextCorrectionView` 硬编码 `activity-ledger/{id}/...`）

## 12. 设计决策记录（ADR）

### 12.1 为什么不直接让招商动态调用活动台账的路由？

- **模块耦合**：招商动态依赖活动台账路由会导致两模块生命周期绑定
- **ID 语义**：activity_id ≠ ledger_id，映射复杂
- **独立演化**：两模块未来可能有不同的音频处理需求

### 12.2 为什么用字段注册表而非直接 import 模型类？

- **解耦**：service 层不依赖任何具体模型
- **扩展性**：新模型只需注册字段映射，无需改 service 代码
- **字段名一致性**：当前 `ActivityLedger` 和 `InvestmentActivity` 字段名相同，但注册表为未来差异化预留了空间

### 12.3 为什么不直接在旧 `audio_processing.py` 上扩展？

- `audio_processing.py` 的 worker 函数签名已固定（`run_async_processing(app, item_id, model_class)`），与 `audio_service` 的 `(app, model_class, item_id)` 不一致，内部仍用 `item_class`（未定义，NameError）
- `register_cancel(item_id)` 缺少模型类键，导致不同模型的取消事件会相互污染
- 新 `audio_service.py` 更清晰地分离了"通用能力"与"模型无关"设计（字段注册表）
- 招商动态改造完成后 `audio_processing.py` 已无任何引用，已删除；统一以 `audio_service.py` 为唯一 worker 入口，避免双份实现分叉
