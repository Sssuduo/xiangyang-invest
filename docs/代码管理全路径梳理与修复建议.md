# 代码管理全路径梳理与修复建议（V16.6）

> 梳理时间：2026-07-21
> 范围：本地开发 → Git（GitHub）→ 生产部署脚本（`deploy.sh`）→ 运行态
> 目标：找出会"悄悄把修复弄丢"的链路问题，并给出可落地的修复意见。

---

## 一、代码管理全路径（现状）

```
[本地开发机 Windows]
   │ 改代码
   ▼
[Git 仓库]  origin = git@github.com:Sssuduo/xiangyang-invest.git
   │  • 分支：master / dev / worktree-*
   │  • 约束：pre-push hook 禁止直推 master，必须走 dev → merge
   │  • 推送通道：Git for Windows 自带的 ssh，但本机实际调用的是
   │            C:\Windows\System32\OpenSSH\ssh.exe（旧版，KEX 不兼容）
   │            → 已在本仓库配置 core.sshCommand 指向系统 ssh 修复
   ▼
[生产服务器 123.56.9.243  /www/wwwroot/invest-app]
   │  ./deploy.sh [分支]
   │    1. 备份 db + 代码
   │    2. git fetch origin
   │    3. git checkout <分支>
   │    4. git reset --hard origin/<分支>   ★ 强制对齐远端
   │    5. pip install / npm install / npm run build
   │    6. rsync -a --delete dist/ → static/   ★ 同步前端产物
   │    7. systemctl restart invest-app + nginx
   │    8. 幂等修正 nginx /static/uploads alias
   ▼
[运行态] Flask + SQLite + Nginx + 前端静态资源
```

**关键认知**：`deploy.sh` 本身是被 Git 跟踪的文件，但它第 4 步执行 `git reset --hard origin/<分支>`，
会把**生产工作区强制还原成 GitHub 上的版本**——包括它自己。这意味着：

> 任何只改在生产磁盘、没进 GitHub 的修复，下一次 `./deploy.sh` 都会被脚本自我抹除。

## 二、已经暴露的问题点

### 问题 1（已发生，最严重）：静态地图数据被部署脚本清空
- **现象**：`deploy.sh` 第 6 步 `rsync -a --delete dist/ static/` 会删除 `static/data/*.json`
  （省份/城市地图数据，由 Git 跟踪但不在 `frontend/dist` 中），导致 `ChinaMap.vue` 的 fetch 全部 404。
- **临时修复**（已落到生产磁盘，未进 Git）：在 `rsync` 行加 `--exclude='data'`，`cp` 回退分支加 `! -name 'data'`。
- **风险**：因为没进 GitHub，下次任意一次部署，`git reset --hard` 会把 `deploy.sh` 还原成旧版，
  修复立即失效，`static/data/` 再次被清空，地图再次 404。

### 问题 2：生产"带外修改"与 Git 不同步（配置漂移）
- 历史上为修 nginx `/static/uploads/` 别名，在生产直接改了 `invest-app.conf` 并用 `chattr +i` 锁死。
- 该配置文件**不在 Git 仓库里**，部署脚本只做"幂等修正"而无法从源码重建。
- 一旦服务器重置/迁移，这段配置就丢失，且无人能从仓库还原其意图。

### 问题 3：本机 Git 推送通道脆弱
- 系统自带 OpenSSH（`C:\Windows\System32\OpenSSH`）对 `known_hosts` 权限极严格，
  而 PowerShell 新建的文件权限过宽会被拒绝读取，导致 `Host key verification failed`。
- Git 实际调用的是系统 ssh 而非 Git 自带 ssh；两者对 GitHub 的 KEX 算法兼容性相反。
- **已修复**：
  - `ssh-keyscan` 写入 `github.com` 的 ed25519/rsa/ecdsa 到 `~/.ssh/known_hosts`；
  - 本仓库 `git config core.sshCommand "C:/Windows/System32/OpenSSH/ssh.exe"`。
  - 副作用：该配置仅作用于本仓库，撤销命令 `git config --unset core.sshCommand`（撤销后 push 会再次失败）。

### 问题 4：分支约束与本地习惯冲突
- `pre-push` hook 要求改动必须 `dev → merge` 才能进 master，禁止直推。
- 但本地常有"直接在 master 改完就想推"的冲动，导致反复被 hook 拦截、流程混乱、
  甚至出现"改动在 master 未提交、又切到 dev 提交"的错位提交。

### 问题 5：部署脚本职责过重且自引用
- 一个 shell 脚本同时负责：备份、拉代码、装依赖、构建、同步静态、重启服务、修 nginx。
- 其中"同步静态"与"修 nginx"都依赖对运行时特殊目录（uploads/data）的特殊豁免，
  这些豁免一旦漏写就会删数据（参见问题 1）。脚本越重，越容易在迭代中引入这类陷阱。

## 三、修复意见（按优先级）

### ★ P0：把 `deploy.sh` 的 `--exclude='data'` 修复持久化进 GitHub
- 这是当下唯一会"自动复发"的故障。必须让 GitHub 上的 `deploy.sh` 也含该修复。
- 走既定 dev 流程（hook 不允许直推 master）：
  ```bash
  cd h:/项目1
  git checkout dev
  git add scripts/deploy.sh
  git commit -m "fix(deploy): 防止 rsync 删除 static/data 地图 JSON"
  git push origin dev
  git checkout master
  git merge dev --no-ff -m "Merge dev: V16.6 deploy.sh 保护 static/data 地图数据"
  git push origin master
  ```
- 同步把生产 `deploy.sh` 重新 scp 覆盖一次（确保运行态与 GitHub 一致，消除当前"仅磁盘有效"的窗口）。

### P1：把"带外配置"纳入 Git 或固化成幂等脚本
- 将 `invest-app.conf` 的 nginx 片段抽成仓库内模板（如 `deploy/nginx/invest-app.conf`），
  由 `deploy.sh` 在每次部署时 `diff`/覆盖并 `nginx -t` 校验，删除 `chattr +i` 的人工锁。
- 或用 `include` 把 `/static/uploads/` 别名独立成 `conf.d/uploads-alias.conf` 纳入仓库。
- 目标：**服务器重置后，从 Git 即可重建全部配置，不依赖任何人工带外改动**。

### P2：固化本机 Git 推送环境
- 方案一（推荐）：把 `core.sshCommand` 写入**全局**或写进仓库随附的 `scripts/setup-ssh.ps1`，
  新人 clone 后一键配置，避免再次踩 host key 坑。
- 方案二：统一使用 Git 自带 ssh（`git install` 时勾选，或在 `~/.ssh/config` 写 `Host github.com` 指定 `IdentityFile` + `User git`），绕开系统 ssh。
- 不建议：长期依赖手动 `ssh-keyscan` 临时修。

### P3：规范分支工作流，减少"错位提交"
- 在仓库根加 `CONTRIBUTING.md` / `docs/分支与部署规范.md`，把"dev → merge → master → deploy"写死成文档。
- 把 `pre-push` hook 的拦截提示写得更具操作性（直接打印上面的 dev 流程命令），降低被拦后的混乱。
- 考虑在 `deploy.sh` 顶部加注释：本脚本会被自身 `git reset --hard` 还原，**任何对它的修改都必须先合进 GitHub**。

### P4：降低部署脚本的"删数据"面
- 将 `static/data`、`static/uploads` 等"运行时持久数据"从 `rsync --delete` 的作用域中彻底隔离：
  例如构建产物只进 `static/dist/`，再由 nginx 把 `static/dist` 作为根，
  `data`/`uploads` 作为独立 location，从源头避免 `--delete` 误伤。
- 或在 `deploy.sh` 里对 `static/data`、`static/uploads` 增加"部署前存在性断言"，
  若缺失则 abort 部署并告警，而非静默清空。
