# 语音知识库与文本校正系统 - 交互逻辑说明

> 适用版本：V15.2+
> 部署时间：2026-07-15

---

## 一、核心概念

### 1.1 语音知识库（VoiceKnowledgeEntry）

| 字段 | 含义 | 示例 |
|------|------|------|
| original | 语音识别出的错误/谐音词 | "火牌" |
| replacement | 正确的替换词 | "伙牌" |
| pinyin | 拼音索引 (自动) | ["huo", "pai"] |
| context | 上下文线索 (可选) | "伙牌镇" |
| usage_count | 被应用次数 | 42 |
| confidence | 置信度 (0~1) | 0.95 |
| source | 来源 | manual / auto_detected |

**匹配逻辑**：

```
输入文本: "襄州区火牌镇召开项目会议"

Step 1: 滑动窗口分词 (2~4字)
  → ["襄州", "州区", "区火", "火牌", "牌镇", ...]

Step 2: 对每个词查知识库拼音索引
  → "火牌" pinyin = "huopai"
  → 匹配到知识库条目: original="火牌" → replacement="伙牌"

Step 3: 计算置信度
  → 基础: 0.85 (知识库条目置信度)
  → 上下文加权: 文本中包含"伙牌镇" → +0.05 = 0.90
  → 频次加权: usage_count > 10 → +0.03 = 0.93

Step 4: 按置信度分级
  → 高置信 (≥0.90): 自动替换
  → 中置信 (0.75~0.89): 提示用户确认
  → 低置信 (<0.75): 仅高亮标记，不替换
```

### 1.2 置信度分级策略

| 级别 | 阈值 | 处理方式 |
|------|------|----------|
| **高** | ≥ 0.90 | 自动替换，无需用户确认 |
| **中** | 0.75 ~ 0.89 | 弹出确认框，用户选择是否替换 |
| **低** | < 0.75 | 黄色高亮，不替换，仅提示 |

**上下文消歧**：
- "火牌" + 上下文"伙牌镇在文本中" → **替换为"伙牌"**
- "火车" + 上下文"高铁站" → **不替换** (不匹配知识库上下文)

### 1.3 文本校正记录 (TextCorrectionRecord)

每次用户在 Web 页执行替换时，记录：
- original_text（原词）
- replacement_text（替换词）
- context_before / context_after（前后各50字）
- method（manual / homophone_auto / homophone_manual）
- confidence（AI推荐置信度）
- persisted_to_knowledge（是否已沉淀到知识库）

**详细操作流程**：

1. **打开校正页**
   - 点击"文本校正" → 新标签页打开 `#/admin/text-correction/{id}`
   - 页面加载台账的 transcript_segmented / transcript_clean / summary_structured

2. **浏览与编辑**
   - 切换 Tab 查看不同版本
   - 鼠标选中词汇（3~30 字符有效）
   - 弹出替换浮层，显示知识库建议
   - 输入替换词 → 点击"替换" → 即时更新页面文本
   - 替换记录自动添加到右侧"校正记录"列表

3. **智能检测**（可选）
   - 点击"检测同音词" → 扫描当前文本
   - 右侧显示候选列表（带置信度分级）
   - 高置信度（≥90%）可直接批量应用
   - 中置信度需逐条确认

4. **保存操作**
   - **应用到台账**：仅保存校正后的文本字段（segmented/clean/summary）
   - **保存并沉淀到知识库**：保存 + 将新校正自动加入知识库

5. **返回原页**
   - 点击"返回" → 关闭标签页 → 回到原始录音抽屉

### 场景 3: 自动同音词校正

```
入口: Web 校正页 → [检测同音词] 按钮
```

**后端执行流程**：

```
POST /api/admin/voice-knowledge/activity-ledger/{id}/auto-correct
Body: { "min_confidence": 0.90 }

↓ 后端逻辑

1. 读取台账 audio_transcript_segmented (分段原文)
2. VoiceKnowledgeService.detect_homophones(text, min_confidence=0.70)
   → 返回候选列表 [{ start, end, source, target, confidence }]
3. 过滤 confidence ≥ 0.90 的候选 → 自动替换
4. 过滤 0.70 ≤ confidence < 0.90 的候选 → 返回给前端提示确认
5. 更新 audio_transcript_segmented = corrected_text
6. 返回 { applied: [...], candidates: [...] }
```

### 场景 4: 术语管理（映射替换）

**与语音知识库的区别**：

| 维度 | TermCorrection | VoiceKnowledgeEntry |

| 匹配方式 | 精确字符串匹配 | 拼音匹配（可模糊） |
| 触发时机 | 手动 / 应用按钮 | ASR 后自动 / Web 校正 |
| 适用范围 | 全部字段 | 主要用于转写文本 |


## 三、关键技术点

### 3.1 拼音索引生成

```python
# pypinyin 无声调输出
lazy_pinyin("火牌")  → ["huo", "pai"]
lazy_pinyin("伙牌")  → ["huo", "pai"]  # 同音！

# 用于建立索引
pinyin_key = "huopai"  # 拼接后做索引键
```

### 3.2 滑动窗口分词

```python
def sliding_tokens(text, max_len=4):
    """生成 2~max_len 字的滑动窗口"""
    results = []
    for size in range(2, max_len + 1):
        for i in range(len(text) - size + 1):
            results.append((i, i + size, text[i:i+size]))
    return results

# "火牌镇" → [("火", "牌", "镇"), ("火牌", "牌镇"), ("火牌镇",)]
```

### 3.3 替换安全策略

| 策略 | 说明 |
|------|------|
| 长短语优先 | 4字短语先匹配，避免"火车"+"站"被错误匹配 |
| 位置去重 | 已匹配的位置不再参与后续匹配 |
| 上下文校验 | 知识库 context 字段存在时，检查文本中是否出现 |
| 频次反馈 | usage_count 越高，置信度加权越高 |
| 分级执行 | 高置信自动、中置信确认、低置信仅标记 |

### 3.4 知识沉淀 (Feedback Loop)

```
用户在 Web 页执行替换
    ↓
记录到 TextCorrectionRecord
    ↓
点击 [保存并沉淀到知识库]
    ↓
持久化到 VoiceKnowledgeEntry
    ↓
usage_count = 1, source = 'auto_detected'
    ↓
下次同音检测时，该知识参与匹配
    ↓
匹配成功 → usage_count += 1
    ↓
频次越高 → 置信度加权越高 → 自动替换比例提升
```

## 五、常见问题

| 问题 | 回答 |
|------|------|
| 为什么"火牌"没有自动替换？ | 置信度可能低于 0.90；尝试 [检测同音词] 手动确认 |
| 中置信度替换怎么操作？ | 检测后列表中"中置信"标签的条目会弹出确认框 |
| 知识条目的 usage_count 有什么用？ | 越高说明该知识越可靠，自动匹配时优先加权 |
| "auto_detected" 和 "manual" 来源有什么区别？ | manual 是人工添加（高置信），auto_detected 是用户校正沉淀（初始较低置信） |
| 替换后想恢复怎么办？ | 当前版本暂无 undo 功能；建议先"应用到台账"前做好备份 |
| 为什么新添加的知识库没有生效？ | 确认 is_active=True；新添加的条目在下一次 detect/update 时生效 |
